# 18. Arsitektur Sistem Tingkat Tinggi

## 18.1 Gambaran Umum
e-Mubtadi'at merupakan aplikasi akademik terintegrasi yang dibangun menggunakan arsitektur modern berbasis **Cloud Native**. Sistem dirancang agar mudah dikembangkan, aman, memiliki performa tinggi, dan mampu menangani pertumbuhan jumlah pengguna tanpa perubahan besar pada arsitektur.

Sistem terdiri atas tiga lapisan utama:
1. **Presentation Layer (Frontend)**
2. **Application Layer (Backend/API)**
3. **Data Layer (Database)**

Seluruh komunikasi antara aplikasi dan backend dilakukan menggunakan REST API melalui protokol HTTPS.

# 19. Prinsip Arsitektur

Seluruh pengembangan aplikasi wajib mengikuti prinsip berikut:

### 19.1 Modular

Setiap modul berdiri sendiri sehingga perubahan pada satu modul tidak memengaruhi modul lain.

> **Contoh:** Modul Penilaian, Modul Dashboard, Modul Santri, dan Modul Laporan dikembangkan secara independen.

### 19.2 Loose Coupling

Ketergantungan antar modul dibuat seminimal mungkin. Komunikasi dilakukan secara ketat melalui **Service Layer** dan **Repository Layer**.

### 19.3 High Cohesion

Setiap modul hanya memiliki satu tanggung jawab utama (*Single Responsibility Principle*).

> **Sebagai contoh:** Modul Penilaian hanya bertanggung jawab terhadap proses penilaian. Modul ini tidak menangani autentikasi maupun manajemen pengguna.

### 19.4 Separation of Concerns (SoC)

Lapisan aplikasi dipisahkan secara hierarkis menjadi:


$$\text{UI} \rightarrow \text{State Management} \rightarrow \text{Service} \rightarrow \text{Repository} \rightarrow \text{API} \rightarrow \text{Database}$$


Tujuannya agar struktur kode rapi dan mudah dipelihara (*maintainable*).

# 21. Prinsip Frontend Flutter

Aplikasi Flutter harus memenuhi standar-standar berikut:

* **Responsive Design:** Tampilan mampu menyesuaikan ukuran layar secara adaptif mulai dari perangkat Android, Tablet, hingga layar monitor Windows Desktop.
* **Reusable Widget:** Komponen UI yang digunakan berulang harus dipisahkan menjadi komponen mandiri.
> **Contoh:** `PrimaryButton`, `PrimaryTextField`, `CardNilai`, `DialogKonfirmasi`, `DataTableAkademik`.


* **Clean Navigation:** Mekanisme perpindahan halaman wajib menggunakan *declarative routing* yang konsisten.
* **Offline Ready:** Data akademik krusial disimpan sementara secara lokal (*local caching*) ketika koneksi internet tidak tersedia, dan disinkronisasikan otomatis saat jaringan pulih.

# 22. Prinsip Backend

Backend dibangun sebagai REST API. Setiap *endpoint* yang berada pada Hono Framework wajib menerapkan layer proteksi berupa:

1. **Authentication:** Memvalidasi keabsahan token pengguna.
2. **Authorization:** Memeriksa hak akses role berdasarkan aturan RBAC.
3. **Validation:** Melakukan sanitasi dan validasi *payload* data masuk.
4. **Error Handling:** Penanganan *exception* secara terpusat untuk menghindari kebocoran log *stack trace* server.
5. **Logging:** Mencatat aktivitas transaksi krusial ke dalam Audit Log.

> 🚫 **PEMBERITAHUAN:** Lapisan Controller/Endpoint dilarang keras melakukan kueri langsung (*raw query*) ke database tanpa melewati *Service Layer* terlebih dahulu.

---

# 23. Prinsip Database

Database menjadi sumber data utama (*Single Source of Truth*). Seluruh perubahan data akademik dilakukan melalui API.

* **Normalisasi Data:** Seluruh struktur tabel minimal memenuhi standar *Third Normal Form* (3NF) guna menghindari anomali data.
* **Integritas Referensial:** Penggunaan *Foreign Key* dan *Constraints* bersifat wajib pada seluruh relasi tabel utama demi menjaga konsistensi data.
* **Soft Delete:** Penghapusan data master dan data akademik yang bersifat krusial wajib menggunakan mekanisme *soft delete* (menggunakan kolom `deleted_at`). Data tidak langsung dihapus dari disk fisik.
* **Audit Trail:** Setiap mutasi data penting (INSERT, UPDATE, DELETE) harus memicu pencatatan otomatis pada tabel `audit_log`.

# 24. Standar API

Seluruh API mengikuti standar REST dengan kata kerja HTTP yang jelas: `GET`, `POST`, `PUT`, `PATCH`, `DELETE`. Format respon data wajib seragam menggunakan struktur **JSON**.

### Respon Sukses (HTTP Status 200/201)

```json
{
  "success": true,
  "message": "Berhasil",
  "data": {}
}

```

### Respon Gagal (HTTP Status 400/401/404/500)

```json
{
  "success": false,
  "message": "Data tidak ditemukan"
}

```

---

# 25. Versioning API

Setiap rute *endpoint* wajib menggunakan penomoran versi pada URL untuk menjamin *backward compatibility* terhadap aplikasi lama jika terdapat pembaruan sistem yang masif.

* **Format Versi Aktif:** `/api/v1/[endpoint-name]`
* **Format Versi Pembaruan Besar:** `/api/v2/[endpoint-name]`

# 27. Konvensi Waktu

* **Format Penyimpanan:** Seluruh data tanggal dan waktu wajib menggunakan standar format **ISO 8601**.
* **Basis Zona:** Waktu disimpan ke dalam database menggunakan standar zona waktu **UTC (+00:00)**.
* **Presentation Layer:** Aplikasi Flutter bertanggung jawab penuh untuk mengonversi waktu UTC tersebut ke zona waktu lokal masing-masing pengguna saat ditampilkan di layar.
